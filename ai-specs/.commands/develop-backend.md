# Role

Adopt the role based on the implementation plan type:
- If plan references `ln-susc-api-standards.mdc` → Use `.agents/lanacion-api-developer.md`
- If plan references `ln-susc-listener-standards.mdc` → Use `.agents/lanacion-lstnr-developer.md`

# Task

# Backend Development Command

Implement backend features following the implementation plan generated by `/plan-backend-ticket`.

## Input

**Plan File**: $ARGUMENTS (e.g., `@SCRUM-500_backend.md`)

## Role Selection (CRITICAL)

Before implementing, you MUST analyze the plan file to determine the correct agent role:

### Step 1: Read the Implementation Plan

1. Open and read the plan file: `$ARGUMENTS`
2. Locate the **"Standards Reference"** or **"Architecture Context"** section
3. Identify which standards document is referenced

### Step 2: Detect Backend Type and Adopt Role

**IF** plan references `ln-susc-api-standards.mdc`:
- **Backend Type**: API (REST endpoints)
- **Adopt Role**: `ai-specs/.agents/lanacion-api-developer.md`
- **Template**: LaNacion.Core.Templates.Web.Api.Minimal
- **Branch Pattern**: `feature/[TICKET-ID]-api`
- **Key Patterns**: Commands/Queries, HTTP endpoints, Event publishing (Outbox)

**ELSE IF** plan references `ln-susc-listener-standards.mdc`:
- **Backend Type**: Listener (SQS event processor)
- **Adopt Role**: `ai-specs/.agents/lanacion-lstnr-developer.md`
- **Template**: ln-SQSlstnr
- **Branch Pattern**: `feature/[TICKET-ID]-listener`
- **Key Patterns**: Event processors, ProcessResult, Idempotency, SQS consumption

**ELSE**:
- **ERROR**: Cannot determine backend type from plan
- **Action**: Ask user to clarify or regenerate plan with `/plan-backend-ticket`

### Step 3: Verify Role Adoption

Once role is determined, confirm:
- ✅ Correct agent role adopted
- ✅ Correct standards document loaded
- ✅ Branch naming pattern matches backend type

## Implementation Process

Follow these steps in order:

### 1. Understand the Plan

- Read the complete implementation plan from `$ARGUMENTS`
- Identify all steps in the "Implementation Steps" section
- Note the "Implementation Order" sequence
- Review "Testing Checklist" and "Implementation Verification"

### 2. Verify/Create Feature Branch

**CRITICAL**: Branch naming MUST match backend type:

```bash
# Check current branch
git branch --show-current

# If not on correct branch:
git checkout main
git pull origin main

# Create branch with correct naming:
# For API:
git checkout -b feature/[TICKET-ID]-api

# For Listener:
git checkout -b feature/[TICKET-ID]-listener
```

**Validation**:
- ✅ Branch name matches backend type from plan
- ✅ Branch follows pattern: `feature/[TICKET-ID]-[api|listener]`
- ❌ Do NOT use generic `feature/[TICKET-ID]` branch

### 3. Implement Step by Step

Execute each step from the plan IN ORDER:

**For Each Step**:
1. Read step description and requirements
2. Identify target files and actions
3. Implement code following the agent's standards:
   - **API Agent**: Follow `ln-susc-api-standards.mdc`
   - **Listener Agent**: Follow `ln-susc-listener-standards.mdc`
4. Verify implementation matches plan specifications
5. Move to next step only when current step is complete

**Key Implementation Rules**:
- Follow the exact file paths specified in the plan
- Use the code examples as templates
- Maintain Clean Architecture layer separation
- Apply CQRS patterns correctly
- Use proper naming conventions

### 4. Implement Tests (Mandatory)

The plan includes a testing step - DO NOT SKIP:

**For APIs**:
- Command/Query handler tests
- Validation tests
- HTTP endpoint tests
- Event publishing tests
- Error handling tests

**For Listeners**:
- Event processor tests
- Idempotency tests
- Validation tests
- Transaction rollback tests
- Error handling tests

**Coverage Target**: 80%+ code coverage

### 5. Run Tests

```bash
# Build solution
dotnet build

# Run all tests
dotnet test

# Verify coverage (if configured)
dotnet test /p:CollectCoverage=true
```

**Validation**:
- ✅ All tests pass
- ✅ No compilation errors
- ✅ Coverage meets 80%+ threshold

### 6. Update Documentation (Mandatory)

The plan includes a documentation update step - DO NOT SKIP:

1. Review all code changes made
2. Identify affected documentation:
   - Data model changes → `ai-specs/specs/data-model.md`
   - API endpoints → `ai-specs/specs/api-spec.yml`
   - Configuration changes → Relevant standards files
3. Update documentation in English
4. Follow `ai-specs/specs/documentation-standards.mdc`

### 7. Code Quality Verification

```bash
# Ensure code passes linting (if configured)
dotnet format --verify-no-changes

# Check for warnings
dotnet build --no-incremental /warnaserror
```

### 8. Commit Changes

**Commit Strategy**:
- Stage ONLY files affected by this ticket
- Leave any unrelated changes unstaged
- Use descriptive commit message

```bash
# Review changes
git status
git diff

# Stage relevant files only
git add [specific-files]

# Commit with descriptive message
git commit -m "feat(TICKET-ID): [description]"

# Examples:
# git commit -m "feat(SCRUM-500): add subscription-created event processor"
# git commit -m "feat(SCRUM-501): add create subscription API endpoint"
```

### 9. Push and Create PR

```bash
# Push branch
git push origin [branch-name]

# Create PR using GitHub CLI
gh pr create --title "[TICKET-ID]: [Feature Description]" \
             --body "Implements [TICKET-ID]\n\nChanges:\n- [list changes]" \
             --base main
```

**PR Requirements**:
- Title includes ticket ID
- Description references ticket
- All tests passing
- Documentation updated

## Backend Type Specific Validations

### For APIs (lanacion-api-developer)

Verify:
- [ ] Commands return specific types (Guid, DTO, etc.)
- [ ] Queries return DTOs
- [ ] Endpoints use REST conventions (plural nouns, no verbs)
- [ ] HTTP status codes correct (200, 201, 204, 400, 404, 500)
- [ ] Events published using Outbox Pattern
- [ ] FluentValidation rules implemented
- [ ] No SQS consumer configuration

### For Listeners (lanacion-lstnr-developer)

Verify:
- [ ] Events implement `IRequest<ProcessResult>`
- [ ] Handlers return `ProcessResult`
- [ ] Idempotency implemented with MensajesRecibidos
- [ ] SQS consumer registered in DI
- [ ] Queue configuration in appsettings.json
- [ ] Event naming follows `evt-{squad}-{entity}-{verb-past}`
- [ ] No HTTP endpoints created

### If Plan File Not Found

```

Please generate the plan first:
1. Run: /plan-backend-ticket [TICKET-ID]
2. Then run: /develop-backend @[TICKET-ID]_backend.md
```

### If Backend Type Cannot Be Determined

```
ERROR: Cannot determine backend type from plan.

The plan must reference either:
- ln-susc-api-standards.mdc (for APIs)
- ln-susc-listener-standards.mdc (for Listeners)

Please regenerate the plan with: /plan-backend-ticket [TICKET-ID]
```

### If Wrong Branch Type

```
WARNING: Branch name does not match backend type.

Expected: feature/[TICKET-ID]-[api|listener]
Current: [current-branch]

Please create correct branch before continuing.
```
